项目主链路

→ → → 低置信度回退 → →

BM25 FAQ 命中流程

用户问题先分词，再与标准问题计算 BM25 得分。Softmax 后最高分超过阈值就直接返回 FAQ 答案。

query -> preprocess -> scores -> softmax -> threshold

Redis 缓存回源

优先读取 Redis 中的问题列表、分词结果和答案；缓存缺失时从 MySQL 回源，并重新写入 Redis。

Redis miss -> MySQL fetch -> Redis set

父子分块

子块负责精确命中，父块负责提供完整上下文，二者通过 parent_id 和 parent_content 关联。

document -> parent chunks -> child chunks

Milvus 混合检索

BGE-M3 同时生成稠密向量和稀疏向量，Milvus 混合检索后再用 CrossEncoder 重排父文档。

dense + sparse -> WeightedRanker -> rerank

术语速查

Redis

内存型键值数据库，本项目用于缓存标准问题、分词结果和已命中的答案，减少重复访问 MySQL。

BM25

关键词相关性算法，用于判断用户问题和 FAQ 标准问题是否足够相似。

Softmax

把一组原始分数转成概率分布，方便用最高分和阈值判断是否命中 FAQ。

Milvus

向量数据库，本项目用于存储知识库文档的 dense/sparse 向量并执行混合检索。

Dense Vector

稠密向量更擅长表达语义相似度，适合召回含义接近但关键词不同的文本。

Sparse Vector

稀疏向量更接近关键词匹配，适合保留专业名词和明确词项的匹配能力。

Rerank

重排序步骤会重新评估 query 和候选父文档的相关性，把更适合生成答案的上下文排到前面。

HyDE

先让 LLM 生成一个假设答案，再用假设答案去检索，适合原问题表达不够接近知识库文本的场景。

BERT 意图识别

判断问题属于通用知识还是专业咨询，决定是否进入 RAG 专业知识检索流程。

WebSocket

用于服务端逐 token 推送 LLM 生成结果，让前端看到流式输出。
