# RAG评估中的四大指标

# 学习目标

学完本节后，你需要掌握三件事：

1. 知道 RAG 评估到底在评什么。
2. 能看懂 RAGAS 评估数据中的四个核心字段。
3. 能根据评估分数判断问题出在“检索阶段”还是“生成阶段”。

---

# 一、先理解：为什么要做 RAG 评估？

RAG 系统不是只靠大模型回答问题，而是先从知识库中检索资料，再把资料交给大模型生成答案。

完整流程如下：

```text
用户问题
   ↓
检索系统从知识库中查找相关资料
   ↓
把用户问题 + 检索资料交给大模型
   ↓
大模型生成最终答案
```

所以，RAG 系统的效果好不好，不能只看最终答案，还要拆开看两个阶段：

```text
第一阶段：检索阶段
问题：资料找得对不对？资料找得全不全？

第二阶段：生成阶段
问题：答案有没有根据资料回答？答案有没有回答用户问题？
```

RAG 评估的核心目的不是简单打一个分数，而是找出系统问题到底发生在哪里：

```text
如果检索阶段有问题，就优化检索。
如果生成阶段有问题，就优化 Prompt 或大模型调用方式。
```

# 二、用一个企业知识库例子理解 RAG 评估

假设我们有一个企业内部知识库，里面存放员工手册、请假制度、报销制度等文档。

用户提出问题：

```text
员工如何申请年假？
```

知识库中有一段正确资料：

```text
员工申请年假时，需要在 OA 系统中提交年假申请，填写请假时间和请假原因。
申请提交后，由直属主管审批。审批通过后，系统会自动同步到考勤系统。
```

一个比较好的 RAG 系统应该这样工作：

```text
用户问题：员工如何申请年假？

检索到的资料：
员工申请年假时，需要在 OA 系统中提交年假申请，填写请假时间和请假原因。
申请提交后，由直属主管审批。审批通过后，系统会自动同步到考勤系统。

生成答案：
员工申请年假时，需要先在 OA 系统中提交年假申请，填写请假时间和请假原因；提交后由直属主管审批，审批通过后会同步到考勤系统。
```

这个结果比较好，因为：

```text
1. 检索系统找到了和年假申请相关的资料。
2. 检索资料覆盖了申请入口、填写内容、审批流程和考勤同步。
3. 大模型生成答案时没有脱离资料乱编。
4. 答案正面回答了“如何申请年假”这个问题。
```

# 三、RAG 评估主要评什么？

RAG 评估可以拆成两大类：

```text
检索评估：评 retrieved_contexts
生成评估：评 response
```

也就是说：

```text
检索评估关注：系统找回来的资料质量。
生成评估关注：大模型最终生成的答案质量。
```

常见的四个核心指标如下：

| 指标 | 评估对象 | 核心问题 | 低分说明 |
|---|---|---|---|
| `context_precision` | 检索结果 | 找到的资料准不准？ | 检索结果里混入了无关资料 |
| `context_recall` | 检索结果 | 需要的资料找全了吗？ | 关键资料没有被检索出来 |
| `faithfulness` | 生成答案 | 答案有没有根据资料回答？ | 大模型出现幻觉或乱编 |
| `answer_relevancy` | 生成答案 | 答案有没有回答用户问题？ | 答案答非所问 |

最简单的记忆方式：

```text
context_precision ：找得准不准
context_recall：找得全不全
faithfulness：有没有乱编
answer_relevancy：有没有答题
```

# 四、RAGAS 评估数据格式

RAGAS 常用的数据格式包含四个字段：

```python
{
    "user_input": "员工如何申请年假？",
    "retrieved_contexts": [
        "员工申请年假时，需要在 OA 系统中提交年假申请。",
        "年假申请需要填写请假时间和请假原因。",
        "申请提交后，由直属主管审批，审批通过后同步到考勤系统。"
    ],
    "response": "员工申请年假时，需要在 OA 系统中提交申请，填写请假时间和请假原因，提交后由直属主管审批，审批通过后会同步到考勤系统。",
    "reference": "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"
}
```

这四个字段分别对应 RAG 流程中的不同部分。

| 字段 | 含义 | 来源 |
|---|---|---|
| `user_input` | 用户问题 | 测试集中的问题 |
| `retrieved_contexts` | 检索到的上下文资料 | 向量库、BM25、混合检索返回的文档 |
| `response` | RAG 系统生成的答案 | 大模型最终输出 |
| `reference` | 标准答案 | 人工准备或评估数据集中已有答案 |

---

# 五、字段逐个解释

## 1. `user_input`：用户问题

```python
"user_input": "员工如何申请年假？"
```

它表示用户真正输入给 RAG 系统的问题。

在项目中，它通常对应代码里的：

```python
query = "员工如何申请年假？"
```

---

## 2. `retrieved_contexts`：检索出来的资料

```python
"retrieved_contexts": [
    "员工申请年假时，需要在 OA 系统中提交年假申请。",
    "年假申请需要填写请假时间和请假原因。",
    "申请提交后，由直属主管审批，审批通过后同步到考勤系统。"
]
```

它表示 RAG 系统从知识库中检索出来的资料。

注意：`retrieved_contexts` 必须是列表。

正确写法：

```python
"retrieved_contexts": [
    "文档片段1",
    "文档片段2",
    "文档片段3"
]
```

即使只检索到一段资料，也要写成列表：

```python
"retrieved_contexts": [
    "员工申请年假时，需要在 OA 系统中提交年假申请。"
]
```

不能写成字符串：

```python
"retrieved_contexts": "员工申请年假时，需要在 OA 系统中提交年假申请。"
```

因为 RAG 系统一次通常会返回多个文档片段，所以 RAGAS 要求它必须是列表格式。

---

## 3. `response`：系统生成答案

```python
"response": "员工申请年假时，需要在 OA 系统中提交申请，填写请假时间和请假原因，提交后由直属主管审批，审批通过后会同步到考勤系统。"
```

它表示 RAG 系统最终生成给用户看的答案。

在项目中，它一般来自：

```python
answer = rag_system.generate_answer(query)
```

也就是大模型最终回答的内容。

---

## 4. `reference`：标准答案

```python
"reference": "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"
```

它表示这道测试题的标准答案。

标准答案不是大模型生成的，而是评估前提前准备好的。

它的作用是给 RAGAS 一个参考标准，用来判断：

```text
系统答案是否正确？
检索资料是否覆盖了标准答案中的关键信息？
```

# 六、代入一条数据，看四个指标怎么判断

假设有一条评估数据：

```python
sample = {
    "user_input": "员工如何申请年假？",
    "retrieved_contexts": [
        "员工申请年假时，需要在 OA 系统中提交年假申请。",
        "年假申请需要填写请假时间和请假原因。",
        "员工报销差旅费时，需要上传发票和行程单。"
    ],
    "response": "员工申请年假时，需要在 OA 系统中提交年假申请，并填写请假时间和请假原因。",
    "reference": "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"
}

user_input：用户问题
retrieved_contexts：从知识库检索到的上下文
response：RAG系统返回的结果
reference：人工标注的标准答案
```

下面分别看四个指标。

---

## 1. 看 `context_precision`

`context_precision` 关注：检索出来的资料准不准。

用户问题是：

```text
员工如何申请年假？
```

检索结果是：

```text
资料1：员工申请年假时，需要在 OA 系统中提交年假申请。        相关
资料2：年假申请需要填写请假时间和请假原因。                  相关
资料3：员工报销差旅费时，需要上传发票和行程单。              无关
```

三条资料中有两条相关，一条无关。

所以可以判断：

```text
context_precision 不算特别高。
```

原因是检索结果中混入了“差旅费报销”这种无关资料。

如果这个指标低，说明检索结果不够精准。

常见优化方向：

```powershell
1. 使用 rerank 对检索结果重新排序。
2. 减少 top_k，避免拿太多无关文档。
3. 优化向量模型。
4. 增加 source_filter，只检索指定知识库或指定文档类型。
5. 清洗知识库中的重复、脏数据。
```

---

## 2. 看 `context_recall`

`context_recall` 关注：标准答案需要的信息，检索结果有没有找全。

标准答案是：

```text
员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。
```

标准答案包含四个关键信息：

```text
1. 在 OA 系统提交申请
2. 填写请假时间和请假原因
3. 直属主管审批
4. 同步到考勤系统
```

检索结果中找到了：

```text
1. 在 OA 系统提交申请
2. 填写请假时间和请假原因
```

没有找到：

```text
3. 直属主管审批
4. 同步到考勤系统
```

所以可以判断：

```text
context_recall 较低。
```

原因是关键资料没有找全。

如果这个指标低，说明检索召回能力不足。

常见优化方向：

```powershell
1. 增大 top_k，让系统多召回一些候选文档。
2. 调整 chunk_size 和 chunk_overlap，避免关键内容被切断。
3. 使用 BM25 + 向量检索的混合检索。
4. 使用 HyDE、子查询、问题改写等方式增强查询。
5. 检查知识库中是否真的包含相关资料。
```

---

## 3. 看 `faithfulness`

`faithfulness` 关注：答案有没有根据检索资料回答，有没有乱编。

系统答案是：

```text
员工申请年假时，需要在 OA 系统中提交年假申请，并填写请假时间和请假原因。
```

检索资料中确实包含：

```text
员工申请年假时，需要在 OA 系统中提交年假申请。
年假申请需要填写请假时间和请假原因。
```

所以系统答案中的信息都能被检索资料支持。

可以判断：

```text
faithfulness 较高。
```

注意：这个答案虽然不完整，但没有乱编。

这也是很多学生容易混淆的地方：

```text
答案不完整，不一定代表 faithfulness 低。
只要答案中的内容都来自检索资料，它的 faithfulness 可能仍然较高。
```

如果 `faithfulness` 低，说明大模型生成了检索资料中没有的信息。

常见优化方向：

```powershell
1. 修改 Prompt，明确要求“只能根据上下文回答”。
2. 没有资料时，要求模型回答“知识库中未找到相关信息”。
3. 降低 temperature，减少自由发挥。
4. 要求答案引用上下文来源。
5. 增加答案校验模块，检查答案是否被上下文支持。
```

---

## 4. 看 `answer_relevancy`

`answer_relevancy` 关注：答案有没有回答用户问题。

用户问：

```text
员工如何申请年假？
```

系统回答：

```text
员工申请年假时，需要在 OA 系统中提交年假申请，并填写请假时间和请假原因。
```

这个答案确是在回答“如何申请年假”。

所以可以判断：

```text
answer_relevancy 较高。
```

但是它没有回答完整，因为缺少主管审批和同步考勤系统。

所以更准确地说：

```text
answer_relevancy 高，但完整性不足。
```

如果这个指标低，说明模型答非所问。

常见优化方向：

```powershell
1. 优化 Prompt，要求先理解用户问题再回答。
2. 增加问题改写或意图识别。
3. 优化上下文拼接方式，把最相关资料放在前面。
4. 控制回答格式，要求直接回答用户问题。
```

# 七、再看一条较差的 RAG 结果

下面这条数据就是问题比较明显的情况。

```python
sample = {
    "user_input": "员工如何申请年假？",
    "retrieved_contexts": [
        "员工报销差旅费时，需要上传发票和行程单。",
        "办公用品采购需要提前提交采购申请。",
        "员工入职时需要提交身份证复印件和银行卡信息。"
    ],
    "response": "员工可以向人事部门口头说明后直接休年假。",
    "reference": "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"
}
```

逐项判断：

| 指标 | 判断 | 原因 |
|---|---|---|
| `context_precision` | 低 | 检索内容都是报销、采购、入职，和年假申请无关 |
| `context_recall` | 低 | 标准答案中的 OA 申请、主管审批、考勤同步都没召回 |
| `faithfulness` | 低 | 答案说“口头说明后直接休年假”，检索资料中没有支持 |
| `answer_relevancy` | 中等或偏低 | 形式上提到了年假，但没有正确回答申请流程 |

这条数据说明：

```text
检索阶段和生成阶段都有问题。
```

优化时不能只改 Prompt，还要优先检查检索系统为什么没有找到年假制度相关资料。

# 八、结合 EduRAG 项目理解评估数据怎么来

假设你的 EduRAG 系统调用链是：

```text
generate_answer()
   ↓
retrieve_and_merge()
   ↓
vector_store.hybrid_search_with_rerank()
   ↓
context_docs
   ↓
llm(prompt)
   ↓
answer
```

那么 RAGAS 评估数据可以这样对应：

| RAGAS 字段 | EduRAG 中对应内容 |
|---|---|
| `user_input` | 用户输入的 `query` |
| `retrieved_contexts` | `context_docs` 中的 `doc.page_content` |
| `response` | `generate_answer()` 返回的 `answer` |
| `reference` | 评估数据集中提前准备的标准答案 |

示例代码：

```python
eval_item = {
    "user_input": query,
    "retrieved_contexts": [doc.page_content for doc in context_docs],
    "response": answer,
    "reference": standard_answer
}
```

这就是把一次 RAG 问答过程转换成一条 RAGAS 评估样本。

# 九、为什么不能只看最终答案？

很多初学者会认为：

```text
我直接看答案对不对不就行了吗？
```

这样不够。

因为答案错了，可能有两种完全不同的原因。

---

## 情况一：检索错了

```text
用户问题：员工如何申请年假？

检索结果：
员工报销差旅费时，需要上传发票和行程单。

生成答案：
员工需要上传发票和行程单。
```

这个问题主要出在检索阶段。

大模型拿到的是错误资料，所以生成答案也跟着错。

---

## 情况二：检索对了，但模型乱答

```text
用户问题：员工如何申请年假？

检索结果：
员工申请年假需要在 OA 系统提交申请，由直属主管审批。

生成答案：
员工可以直接休年假，不需要审批。
```

这个问题主要出在生成阶段。

检索资料是对的，但大模型没有严格根据上下文回答。

---

所以 RAG 评估必须拆成两部分看：

```text
检索有没有问题？
生成有没有问题？
```

这就是 RAGAS 评估指标存在的意义。

# 十、不同低分对应什么优化方向？

| 低分指标 | 主要问题 | 优先优化方向 |
|---|---|---|
| `context_precision` 低 | 检索不精准，混入无关内容 | rerank、减少 top_k、优化向量模型、增加过滤条件 |
| `context_recall` 低 | 关键资料没找全 | 增大 top_k、优化分块、混合检索、问题改写 |
| `faithfulness` 低 | 大模型乱编 | 优化 Prompt、降低 temperature、要求基于上下文回答 |
| `answer_relevancy` 低 | 答非所问 | 优化 Prompt、问题改写、意图识别、调整上下文顺序 |

# 十一、RAG 评估的基本流程

完整的 RAG 评估流程可以分为 5 步。

```text
第 1 步：准备测试问题
例如：员工如何申请年假？

第 2 步：准备标准答案
例如：员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。

第 3 步：让 RAG 系统回答问题
得到 response。

第 4 步：保存检索到的文档片段
得到 retrieved_contexts。

第 5 步：使用 RAGAS 计算评估指标
得到 context_precision、context_recall、faithfulness、answer_relevancy 等分数。
```

也可以表示为：

```text
测试问题 + 标准答案
        ↓
运行 RAG 系统
        ↓
得到检索资料和生成答案
        ↓
组织成 RAGAS 数据格式
        ↓
执行评估
        ↓
根据指标定位问题
```

# 十二、最小代码示例：构造一条评估数据

```python
query = "员工如何申请年假？"

retrieved_contexts = [
    "员工申请年假时，需要在 OA 系统中提交年假申请。",
    "年假申请需要填写请假时间和请假原因。",
    "申请提交后，由直属主管审批，审批通过后同步到考勤系统。"
]

response = "员工申请年假时，需要在 OA 系统中提交申请，填写请假时间和请假原因，提交后由直属主管审批，审批通过后会同步到考勤系统。"

reference = "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"

eval_item = {
    "user_input": query,
    "retrieved_contexts": retrieved_contexts,
    "response": response,
    "reference": reference
}

print(eval_item)
```

这段代码的作用不是执行评估，而是先构造一条标准的评估数据。

只有数据格式正确，后面才能交给 RAGAS 执行评估。

---

# 十三、数据格式检查代码

在执行 RAGAS 评估前，建议先检查数据格式。

```python
required_fields = ["user_input", "retrieved_contexts", "response", "reference"]

for index, item in enumerate(data):
    for field in required_fields:
        if field not in item:
            raise ValueError(f"第 {index + 1} 条数据缺少字段：{field}")

    if not isinstance(item["retrieved_contexts"], list):
        raise TypeError(
            f"第 {index + 1} 条数据的 retrieved_contexts 必须是列表，"
            f"例如：['检索到的上下文内容']"
        )
```

这段代码的作用是：

```text
1. 检查每条数据有没有 user_input。
2. 检查每条数据有没有 retrieved_contexts。
3. 检查每条数据有没有 response。
4. 检查每条数据有没有 reference。
5. 检查 retrieved_contexts 是否是列表。
```

它不是 RAGAS 的核心评估代码，而是评估前的数据格式检查代码。

# 十四、最容易混淆的问题

## 1. `response` 和 `reference` 有什么区别？

| 字段 | 含义 |
|---|---|
| `response` | RAG 系统生成的答案 |
| `reference` | 标准答案 |

例子：

```python
"response": "员工可以向人事部门口头说明后直接休年假。"
"reference": "员工申请年假需要在 OA 系统提交申请，经直属主管审批后同步到考勤系统。"
```

这说明系统生成的答案和标准答案不一致。

---

## 2. `retrieved_contexts` 和 `reference` 有什么区别？

| 字段 | 含义 |
|---|---|
| `retrieved_contexts` | 系统从知识库中检索出来的原始资料 |
| `reference` | 人工整理好的标准答案 |

例子：

```python
"retrieved_contexts": [
    "员工申请年假时，需要在 OA 系统中提交年假申请。",
    "申请提交后，由直属主管审批。"
]
```

```python
"reference": "员工申请年假需要在 OA 系统提交申请，经直属主管审批后同步到考勤系统。"
```

`retrieved_contexts` 是资料，`reference` 是答案标准。

---

## 3. 为什么 `retrieved_contexts` 必须是列表？

因为 RAG 系统一次通常会检索多个文档片段。

例如：

```python
"retrieved_contexts": [
    "文档片段1",
    "文档片段2",
    "文档片段3"
]
```

即使只有一个文档片段，也要写成：

```python
"retrieved_contexts": [
    "文档片段1"
]
```

不能写成：

```python
"retrieved_contexts": "文档片段1"
```

# 十五、最小记忆版

RAG 评估只需要先记住两件事。

第一，RAG 评估分两块：

```text
检索评估：资料找得怎么样？
生成评估：答案生成得怎么样？
```

第二，四个核心指标这样记：

```text
context_precision：检索结果准不准
context_recall：关键资料全不全
faithfulness：答案有没有乱编
answer_relevancy：答案有没有答题
```

再记住四个数据字段：

```text
user_input：用户问题
retrieved_contexts：检索资料
response：系统答案
reference：标准答案
```

最后，看到低分时这样判断：

```text
检索指标低：优先优化检索系统。
生成指标低：优先优化 Prompt、大模型参数或答案约束。
```

# 十六、课堂练习

## 练习 1：判断指标问题

给定数据：

```python
sample = {
    "user_input": "员工如何申请年假？",
    "retrieved_contexts": [
        "员工申请年假时，需要在 OA 系统中提交年假申请。",
        "年假申请提交后，由直属主管审批。"
    ],
    "response": "员工申请年假需要在 OA 系统提交申请，并经过直属主管审批。",
    "reference": "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"
}
```

请判断：

```text
1. context_precision 高还是低？为什么？
2. context_recall 高还是低？为什么？
3. faithfulness 高还是低？为什么？
4. answer_relevancy 高还是低？为什么？
```

参考答案：

```text
1. context_precision 较高，因为检索到的两条资料都和年假申请相关。
2. context_recall 中等或偏低，因为缺少“填写请假时间和原因”“同步到考勤系统”等信息。
3. faithfulness 较高，因为答案中的内容都能从检索资料中找到依据。
4. answer_relevancy 较高，因为答案回答了如何申请年假。
```

## 练习 2：判断问题出在哪里

给定数据：

```python
sample = {
    "user_input": "员工如何申请年假？",
    "retrieved_contexts": [
        "员工申请年假时，需要在 OA 系统中提交年假申请，填写请假时间和请假原因，由直属主管审批。"
    ],
    "response": "员工可以直接向同事说明后休年假，不需要在系统中申请。",
    "reference": "员工申请年假需要在 OA 系统提交申请，填写请假时间和原因，经直属主管审批后同步到考勤系统。"
}
```

请判断主要问题出在哪里。

参考答案：

```text
主要问题出在生成阶段。

原因：
检索结果中已经包含“OA 系统提交申请”“填写请假时间和原因”“直属主管审批”等正确信息，说明检索并不差。
但是 response 说“直接向同事说明后休年假，不需要系统申请”，这和检索资料冲突，属于大模型没有根据上下文回答。

所以 faithfulness 会比较低。
```

# 本章小结

RAG 评估的核心不是为了得到一个漂亮分数，而是为了定位问题。

当系统回答不好时，需要判断：

```text
是知识库没有检索到正确资料？
还是资料找到了，但大模型没有正确使用？
```

对应到指标就是：

```text
context_precision、context_recall：看检索质量。
faithfulness、answer_relevancy：看生成质量。
```

只要能看懂这四个指标，就能开始系统性优化 RAG 项目。
