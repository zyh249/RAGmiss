# BM25 算法教程

## 一、学习目标

学完本教程后，需要掌握：

1. BM25 是什么；
2. BM25 为什么适合做关键词检索；
3. BM25 公式中每个参数的含义；
4. IDF 是怎么计算出来的；
5. 如何一步一步手算 BM25 得分；
6. 如何用 Python 实现一个简单 BM25 检索；
7. 如何用快速技巧公式估算 BM25 排序。

---

## 二、BM25 是什么？

BM25 是一种经典的**关键词检索算法**。

它的作用是：

> 用户输入一个问题，BM25 根据关键词匹配情况，给每篇文档打分，分数越高，说明文档越相关。

例如用户查询：

```text
Redis 缓存 过期时间
```

知识库中有 4 条文档：

```text
D1：Redis 设置 缓存 过期时间
D2：Redis 内存 数据库
D3：Python 安装 第三方库
D4：MySQL 创建 数据库
```

BM25 会认为：

```text
D1 最相关
```

因为 D1 同时命中了：

```text
Redis、缓存、过期时间
```

---

## 三、BM25 的核心思想

BM25 可以先记成一句话：

```text
BM25 = 关键词匹配 + 稀有词加权 + 文档长度修正
```

也就是说，BM25 主要看三件事：

| 判断因素 | 含义 |
|---|---|
| 查询词有没有出现 | 出现了就加分 |
| 查询词是不是稀有 | 越少见，权重越高 |
| 文档是不是太长 | 太长会适当降分 |

例如：

```text
Redis、缓存、过期时间
```

这些词如果都出现在一篇文档中，这篇文档通常更相关。

但是：

```text
如何、怎么、使用、方法
```

这些词太常见，即使出现了，区分度也不强。

---

## 四、BM25 真实公式

BM25 常用公式如下：

```text
score(D, Q) = Σ IDF(qi) × [ f(qi,D) × (k1 + 1) ]
                         / [ f(qi,D) + k1 × (1 - b + b × |D| / avgdl) ]
```

第一次看这个公式可能会觉得复杂，但它可以拆成三层理解。

---

## 五、先把 BM25 公式拆开理解

### 1. 最外层：文档总分

```text
文档总分 = 所有查询词得分相加
```

例如用户查询：

```text
Redis 缓存 过期时间
```

那么某篇文档的 BM25 总分就是：

```text
文档总分 = Redis得分 + 缓存得分 + 过期时间得分
```

如果某个词没有出现在文档中，这个词的得分就是 0。

---

### 2. 每个查询词得分

```text
查询词得分 = IDF × 修正值
```

其中：

```text
IDF：这个词有多重要
修正值：这个词在当前文档中匹配得好不好
```

---

### 3. 修正值

BM25 里最复杂的是修正值：

```text
修正值 = [ f × (k1 + 1) ] / [ f + k1 × (1 - b + b × |D| / avgdl) ]
```

它主要做两件事：

```text
1. 看这个词在文档中出现了几次；
2. 看当前文档是不是太长。
```

---

## 六、公式参数说明

| 参数 | 含义 | 通俗解释 |
|---|---|---|
| `score(D, Q)` | 文档 D 对查询 Q 的得分 | 分数越高，文档越相关 |
| `D` | 当前文档 | 比如某一条 FAQ |
| `Q` | 用户查询 | 比如"Redis 缓存 过期时间" |
| `qi` | 查询中的某个词 | 比如 Redis、缓存、过期时间 |
| `Σ` | 求和 | 把每个查询词的得分加起来 |
| `IDF(qi)` | 查询词的重要程度 | 词越少见，IDF 越高 |
| `f(qi,D)` | 查询词在文档 D 中出现次数 | 出现次数越多，通常越相关 |
| `k1` | 词频调节参数 | 控制词出现多次时能加多少分 |
| `b` | 文档长度调节参数 | 控制长文档惩罚强度 |
| `|D|` | 当前文档长度 | 当前文档分词后有多少个词 |
| `avgdl` | 平均文档长度 | 所有文档长度的平均值 |

常用参数：

```text
k1 = 1.5
b = 0.75
```

其中：

```text
k1 越大，词频影响越明显；
b 越大，文档长度影响越明显。
```

---

## 七、IDF 是什么？

IDF 是 BM25 中非常重要的一部分。

它用来衡量：

> 一个词在整个文档库中有多"稀有"、多"重要"。

核心逻辑是：

```text
一个词出现在越少的文档中，它的 IDF 越高；
一个词出现在越多的文档中，它的 IDF 越低。
```

例如查询：

```text
Redis 缓存 过期时间
```

这 3 个词的重要程度不一定一样。

如果一个词在很多文档里都出现，比如：

```text
Redis
```

说明它比较常见，区分能力一般。

如果一个词只在很少文档里出现，比如：

```text
过期时间
```

说明它更具体，区分能力更强。

---

## 八、IDF 的计算公式

BM25 中常用的 IDF 公式是：

```text
IDF(qi) = ln(1 + (N - df(qi) + 0.5) / (df(qi) + 0.5))
```

参数说明：

| 参数 | 含义 |
|---|---|
| `qi` | 查询中的某个词 |
| `N` | 文档总数量 |
| `df(qi)` | 包含这个词的文档数量 |
| `ln` | 自然对数 |
| `0.5` | 平滑参数，避免极端情况 |

---

## 九、IDF 详细计算案例

现在有 4 篇文档：

```text
D1：Redis 设置 缓存 过期时间
D2：Redis 内存 数据库
D3：Python 安装 第三方库
D4：MySQL 创建 数据库
```

所以：

```text
N = 4
```

用户查询：

```text
Redis 缓存 过期时间
```

---

### 1. 计算 Redis 的 IDF

第一步，看 Redis 出现在几篇文档里。

```text
D1：Redis 设置 缓存 过期时间
D2：Redis 内存 数据库
```

Redis 出现在 D1 和 D2 中，所以：

```text
df(Redis) = 2
```

代入公式：

```text
IDF(Redis) = ln(1 + (N - df + 0.5) / (df + 0.5))
```

得到：

```text
IDF(Redis) = ln(1 + (4 - 2 + 0.5) / (2 + 0.5))
```

一步一步算：

```text
4 - 2 + 0.5 = 2.5
2 + 0.5 = 2.5
2.5 / 2.5 = 1
1 + 1 = 2
ln(2) ≈ 0.6931
```

所以：

```text
IDF(Redis) ≈ 0.6931
```

---

### 2. 计算 缓存 的 IDF

缓存只出现在 D1 中。

```text
D1：Redis 设置 缓存 过期时间
```

所以：

```text
df(缓存) = 1
```

代入公式：

```text
IDF(缓存) = ln(1 + (4 - 1 + 0.5) / (1 + 0.5))
```

一步一步算：

```text
4 - 1 + 0.5 = 3.5
1 + 0.5 = 1.5
3.5 / 1.5 ≈ 2.3333
1 + 2.3333 = 3.3333
ln(3.3333) ≈ 1.2040
```

所以：

```text
IDF(缓存) ≈ 1.2040
```

---

### 3. 计算 过期时间 的 IDF

过期时间也只出现在 D1 中。

所以：

```text
df(过期时间) = 1
```

计算方式和"缓存"一样：

```text
IDF(过期时间) = ln(1 + (4 - 1 + 0.5) / (1 + 0.5))
```

所以：

```text
IDF(过期时间) ≈ 1.2040
```

---

### 4. IDF 结果汇总

| 查询词 | 出现在哪些文档 | df | IDF |
|---|---|---:|---:|
| Redis | D1、D2 | 2 | 0.6931 |
| 缓存 | D1 | 1 | 1.2040 |
| 过期时间 | D1 | 1 | 1.2040 |

可以看到：

```text
缓存、过期时间 比 Redis 更重要。
```

因为：

```text
Redis 出现在 2 篇文档中；
缓存 只出现在 1 篇文档中；
过期时间 只出现在 1 篇文档中。
```

所以：

```text
缓存、过期时间 更稀有，更能区分文档主题。
```

## 十一、计算 D1 的 BM25 得分

D1：

```text
Redis 设置 缓存 过期时间
```

D1 长度：

```text
|D1| = 4
```

D1 命中 3 个查询词：

```text
Redis、缓存、过期时间
```

并且每个词都出现 1 次：

```text
f(Redis, D1) = 1
f(缓存, D1) = 1
f(过期时间, D1) = 1
```

---

### 1. 固定参数后的简化公式

当：

```text
k1 = 1.5
b = 0.75
```

原公式中的单词得分可以写成：

```text
单词得分 = IDF × [ f × 2.5 ] / [ f + 1.5 × (0.25 + 0.75 × |D| / avgdl) ]
```

如果某个词只出现 1 次，也就是：

```text
f = 1
```

可以继续简化为：

```text
单词得分 = IDF × 2.5 / [ 1 + 1.5 × (0.25 + 0.75 × |D| / avgdl) ]
```

---

### 2. 计算 D1 的修正值

先计算修正值：

```text
修正值 = 2.5 / [1 + 1.5 × (0.25 + 0.75 × |D1| / avgdl)]
```

代入：

```text
|D1| = 4
avgdl = 3.25
```

得到：

```text
修正值 = 2.5 / [1 + 1.5 × (0.25 + 0.75 × 4 / 3.25)]
```

一步一步算：

```text
4 / 3.25 = 1.2308
0.75 × 1.2308 = 0.9231
0.25 + 0.9231 = 1.1731
1.5 × 1.1731 = 1.7596
1 + 1.7596 = 2.7596
2.5 / 2.7596 ≈ 0.9059
```

所以：

```text
D1 的修正值 ≈ 0.9059
```

---

### 3. 计算 D1 每个词的得分

Redis 得分：

```text
Redis得分 = 0.6931 × 0.9059
          ≈ 0.6279
```

缓存得分：

```text
缓存得分 = 1.2040 × 0.9059
         ≈ 1.0907
```

过期时间得分：

```text
过期时间得分 = 1.2040 × 0.9059
             ≈ 1.0907
```

---

### 4. 计算 D1 总分

```text
D1总分 = Redis得分 + 缓存得分 + 过期时间得分
```

代入：

```text
D1总分 = 0.6279 + 1.0907 + 1.0907
```

结果：

```text
D1总分 = 2.8093
```

所以：

```text
D1 BM25得分 ≈ 2.8093
```

## 十五、为什么 D1 比 D2 分数高？

D2 只命中了：

```text
Redis
```

D1 命中了：

```text
Redis、缓存、过期时间
```

而且：

```text
缓存、过期时间 的 IDF 更高
```

所以 D1 的总分远高于 D2。

可以通俗理解为：

```text
D2：只知道用户在问 Redis；
D1：知道用户在问 Redis 的缓存过期时间。
```

所以 D1 更相关。

## 十六、BM25 不等于简单数关键词

BM25 不是简单地数关键词出现次数。

例如：

```text
D5：Redis Redis Redis 缓存
```

虽然 Redis 出现了 3 次，但 BM25 不会让分数无限增长。

原因是公式中有这一部分：

```text
f(qi,D) × (k1 + 1)
--------------------
f(qi,D) + k1 × (...)
```

这个结构会让词频加分逐渐变慢。

也就是说：

```text
词出现次数越多，分数会增加；
但是增加幅度越来越小，不会无限加分。
```

这比普通关键词计数更加合理。

## 十七、BM25 快速技巧公式

在课堂讲解、面试表达、手工估算时，可以使用一个快速公式帮助理解。

快速公式：

```text
BM25快速分 ≈ 命中词IDF之和 × 长度修正系数
```

也可以写成：

```text
BM25快速分 ≈ 命中词IDF之和 × avgdl / 当前文档长度
```

通俗理解：

```text
先看文档命中了哪些查询词；
把这些词的 IDF 加起来；
再根据文档长度稍微修正一下。
```

---

### 1. 快速计算 D1

D1 命中：

```text
Redis、缓存、过期时间
```

命中词 IDF 之和：

```text
0.6931 + 1.2040 + 1.2040 = 3.1011
```

长度修正：

```text
avgdl / |D1| = 3.25 / 4 = 0.8125
```

D1 快速分：

```text
3.1011 × 0.8125 ≈ 2.5196
```

---

### 2. 快速计算 D2

D2 命中：

```text
Redis
```

命中词 IDF 之和：

```text
0.6931
```

长度修正：

```text
avgdl / |D2| = 3.25 / 3 ≈ 1.0833
```

D2 快速分：

```text
0.6931 × 1.0833 ≈ 0.7509
```

---

### 3. 快速排序

| 文档 | 快速分 |
|---|---:|
| D1 | 2.5196 |
| D2 | 0.7509 |
| D3 | 0 |
| D4 | 0 |

快速公式的排序结果仍然是：

```text
D1 > D2 > D3 = D4
```

这个结果和真实 BM25 排序一致。

注意：快速公式不是严格 BM25，只适合理解和估算，不建议用于生产环境。

## 十八、Python 手动验证 BM25 计算

下面代码使用刚才相同的公式，手动计算 BM25 得分。

```python
import math

# 1. 文档分词结果
docs = [
    ["Redis", "设置", "缓存", "过期时间"],
    ["Redis", "内存", "数据库"],
    ["Python", "安装", "第三方库"],
    ["MySQL", "创建", "数据库"]
]

# 2. 查询词
query = ["Redis", "缓存", "过期时间"]

# 3. BM25 参数
k1 = 1.5
b = 0.75

# 4. 文档总数
N = len(docs)

# 5. 平均文档长度
avgdl = sum(len(doc) for doc in docs) / N


# 6. 计算某个词出现在多少篇文档中
def get_df(term):
    count = 0

    for doc in docs:
        if term in doc:
            count += 1

    return count


# 7. 计算 IDF
def get_idf(term):
    df = get_df(term)
    idf = math.log(1 + (N - df + 0.5) / (df + 0.5))
    return idf


# 8. 计算单篇文档的 BM25 得分
def bm25_score(doc, query):
    score = 0
    doc_len = len(doc)

    for term in query:
        # 查询词在当前文档中出现的次数
        f = doc.count(term)

        # 如果没有出现，该词得分为 0
        if f == 0:
            continue

        # 计算 IDF
        idf = get_idf(term)

        # 计算 BM25 公式中的分子
        numerator = f * (k1 + 1)

        # 计算 BM25 公式中的分母
        denominator = f + k1 * (1 - b + b * doc_len / avgdl)

        # 当前词的 BM25 得分
        term_score = idf * numerator / denominator

        # 累加到文档总分
        score += term_score

    return score


# 9. 输出结果
for i, doc in enumerate(docs, start=1):
    score = bm25_score(doc, query)
    print(f"D{i}: {doc}, BM25得分 = {score:.4f}")
```

运行结果：

```text
D1: ['Redis', '设置', '缓存', '过期时间'], BM25得分 = 2.8093
D2: ['Redis', '内存', '数据库'], BM25得分 = 0.7179
D3: ['Python', '安装', '第三方库'], BM25得分 = 0.0000
D4: ['MySQL', '创建', '数据库'], BM25得分 = 0.0000
```

这和前面的手算结果一致。

## 十九、使用 rank-bm25 实现检索

实际项目中，一般不会自己手写 BM25 公式，而是使用 `rank-bm25` 库。

安装依赖：

```bash
pip install rank-bm25 jieba
```

完整代码：

```python
from rank_bm25 import BM25Okapi
import jieba
import numpy as np

# 1. 准备文档
documents = [
    "Redis 设置 缓存 过期时间",
    "Redis 内存 数据库",
    "Python 安装 第三方库",
    "MySQL 创建 数据库"
]

# 2. 文档分词
tokenized_documents = [list(jieba.cut(doc)) for doc in documents]

# 3. 创建 BM25 模型
bm25 = BM25Okapi(tokenized_documents)

# 4. 用户查询
query = "Redis 缓存 过期时间"
tokenized_query = list(jieba.cut(query))

# 5. 计算 BM25 得分
scores = bm25.get_scores(tokenized_query)

# 6. 按得分从高到低排序
ranked_indices = np.argsort(scores)[::-1]

# 7. 输出结果
for index in ranked_indices:
    print("文档：", documents[index])
    print("BM25得分：", scores[index])
    print("-" * 30)
```

```text

tokenized_documents = [list(jieba.cut(doc)) for doc in documents]

ranked_indices = np.argsort(scores)[::-1]
```

## 二十一、k1 和 b 到底怎么理解？

在 BM25 中，`k1` 和 `b` 是两个非常重要的调节参数。

很多同学容易误解为：

```text
k1 = 1.5，是不是表示出现 1.5 个词才加分？
b = 0.75，是不是表示文档超过 0.75 倍长度才惩罚？
```

这个理解是不准确的。

更准确地说：

```text
k1：控制"同一个查询词重复出现时，加分增长得有多快"
b：控制"当前文档长度偏离平均长度时，修正力度有多强"
```

---

### 1. k1 不是"多少个词会加分"

`k1` 不表示"多少个词会加分"。

只要查询词在文档中出现了，就会参与计算。

例如用户查询：

```text
Redis 缓存
```

文档：

```text
D1：Redis 缓存
D2：Redis Redis Redis 缓存
```

D1 和 D2 都命中了 `Redis`，都会加分。

区别在于：

```text
D2 中 Redis 出现次数更多，所以 Redis 这部分得分会更高；
但是 BM25 不会让 Redis 因为重复很多次就无限加分。
```

这就是 `k1` 的作用。

---

### 2. k1 控制词频加分的"饱和速度"

先不考虑文档长度，假设当前文档长度刚好等于平均长度。

这时词频修正可以简化为：

```text
词频修正 = f × (k1 + 1) / (f + k1)
```

其中：

```text
f = 查询词在当前文档中出现的次数
```

当：

```text
k1 = 1.5
```

公式变成：

```text
词频修正 = f × 2.5 / (f + 1.5)
```

---

### 3. k1 = 1.5 时，词频怎么影响得分？

| 查询词出现次数 f | 词频修正值 | 说明 |
|---:|---:|---|
| 1 次 | 1.000 | 正常加分 |
| 2 次 | 1.429 | 继续加分，但不是 2 倍 |
| 3 次 | 1.667 | 还会加分，但增长变慢 |
| 5 次 | 1.923 | 加分继续变慢 |
| 10 次 | 2.174 | 已经接近上限 |
| 无限多次 | 2.500 | 最大接近 `k1 + 1` |

所以：

```text
k1 = 1.5 不代表"出现 1.5 次才加分"。
```

它表示：

```text
词出现次数越多，分数会增加；
但增加幅度会越来越小；
最多接近 k1 + 1，也就是 2.5 倍。
```

课堂上可以粗略记成：

```text
k1 = 1.5 时，词出现 1~3 次影响比较明显；
超过 3 次以后，继续重复的加分越来越不明显。
```

---

### 4. b 不是"固定多长开始惩罚"

`b` 也不是一个固定长度。

BM25 不会规定：

```text
超过 100 个词就惩罚
超过 200 个词就惩罚
```

它看的是：

```text
当前文档长度 和 平均文档长度 的比例
```

也就是公式中的：

```text
|D| / avgdl
```

其中：

```text
|D| = 当前文档长度
avgdl = 所有文档的平均长度
```

判断规则是：

```text
|D| > avgdl：当前文档比平均长度长，会被长度惩罚
|D| = avgdl：当前文档等于平均长度，不惩罚也不加成
|D| < avgdl：当前文档比平均长度短，会有一点加成
```

---

### 5. b 控制长度修正的强弱

BM25 中和长度有关的部分是：

```text
1 - b + b × |D| / avgdl
```

当：

```text
b = 0.75
```

公式变成：

```text
长度影响 = 0.25 + 0.75 × |D| / avgdl
```

---

### 6. b = 0.75 时，文档长度怎么影响得分？

假设平均文档长度：

```text
avgdl = 100
```

现在有 4 篇文档：

| 文档 | 当前长度 | 长度影响计算 | 长度影响 |
|---|---:|---|---:|
| D1 | 50 | 0.25 + 0.75 × 50 / 100 | 0.625 |
| D2 | 100 | 0.25 + 0.75 × 100 / 100 | 1.000 |
| D3 | 150 | 0.25 + 0.75 × 150 / 100 | 1.375 |
| D4 | 200 | 0.25 + 0.75 × 200 / 100 | 1.750 |

可以这样理解：

```text
长度影响 < 1：文档偏短，长度修正对得分更友好
长度影响 = 1：文档长度刚好平均，不修正
长度影响 > 1：文档偏长，会被长度惩罚
```

注意：这个"长度影响"出现在 BM25 公式的分母中。

```text
分母越大，最终得分越低。
```

所以：

```text
文档越长，长度影响越大，分母越大，得分会被压低。
```

---

### 7. b = 0、0.75、1 有什么区别？

| b 的取值 | 含义 | 效果 |
|---:|---|---|
| 0 | 不考虑文档长度 | 长文档、短文档不因长度改变分数 |
| 0.75 | 常用折中值 | 考虑长度，但不过度依赖长度 |
| 1 | 完全按长度比例修正 | 文档长度影响最明显 |

所以：

```text
b = 0.75 不是固定阈值，而是长度修正力度。
```

---

### 8. 直接回答两个问题

#### 问题 1：k1 = 1.5 代表多少个词会加分？

不是代表多少个词，而是：

```text
同一个查询词只要出现 1 次就会加分；
出现 2 次、3 次还会继续加分；
但是从 2~3 次以后，加分增长开始明显变慢；
出现很多次，也不会无限加分。
```

更准确地说：

```text
k1 控制词频 f 的饱和速度。
```

当 `k1 = 1.5` 时：

```text
词频修正最大接近 2.5；
也就是重复再多次，也不会无限放大。
```

---

#### 问题 2：b = 0.75 代表文档多长会被惩罚？

不是固定多长，而是：

```text
只要当前文档长度大于平均文档长度，就开始被长度惩罚。
```

例如：

```text
avgdl = 100
```

那么：

```text
文档长度 80：比平均短，略微加成
文档长度 100：等于平均，不惩罚
文档长度 120：比平均长，轻微惩罚
文档长度 200：比平均长很多，惩罚更明显
```

更准确地说：

```text
b 控制的是"超过平均长度以后，惩罚有多强"。
```

### 9. 最小记忆版

```text
k1 管重复词：
同一个查询词出现越多，分数越高；
但 k1 控制它不能无限变高。

b 管文档长度：
文档超过平均长度，就会被惩罚；
文档低于平均长度，会略微加成；
b 控制这种长度修正的强弱。
```

再压缩：

```text
k1 看"同一个词重复几次"。
b 看"当前文档比平均长度长多少"。
```


## 二十二、BM25 和向量检索的区别

| 对比项 | BM25 | 向量检索 |
|---|---|---|
| 检索依据 | 关键词匹配 | 语义相似 |
| 是否需要模型 | 不需要 Embedding 模型 | 需要 Embedding 模型 |
| 适合内容 | 报错、命令、函数名、专业词 | 同义表达、语义问答 |
| 优点 | 快、稳定、可解释 | 能理解语义相似 |
| 缺点 | 不理解同义词 | 对专业词、代码词可能不稳定 |

例如：

```text
用户问：Redis 的 key 怎么自动失效？
文档写：Redis 如何设置缓存过期时间？
```

BM25 可能只能匹配到：

```text
Redis
```

但是向量检索更容易理解：

```text
自动失效 ≈ 过期时间
```

所以在 RAG 系统中，经常使用：

```text
BM25 + 向量检索 + Rerank
```

## 二十三、BM25 适合什么场景？

BM25 特别适合：

```text
FAQ 问答
报错检索
命令检索
API 文档检索
代码片段检索
专业名词检索
短文本问题匹配
```

例如：

```text
ModuleNotFoundError
MilvusClient
Connection refused
Redis expire
BM25Okapi
```

这些关键词明确的内容，BM25 通常效果很好。

## 二十四、BM25 的缺点

BM25 的主要缺点是：

```text
不理解语义，只看关键词。
```

例如下面这些词语含义相近：

```text
过期时间
失效时间
TTL
自动删除
```

BM25 不一定知道它们是一回事。

所以，BM25 适合作为关键词检索基础能力，但在复杂 RAG 项目中，通常会和向量检索结合使用。

## 二十六、最小记忆版

BM25 一句话：

> BM25 是一种关键词相关性打分算法，它综合考虑查询词是否出现、查询词是否稀有、查询词出现次数和文档长度，计算文档相关性分数。

核心公式记忆：

```text
文档得分 = 所有查询词得分相加
```

查询词得分记忆：

```text
单词得分 = IDF × 词频修正 × 文档长度修正
```

最终理解：

```text
BM25 = 关键词匹配 + 稀有词加权 + 文档长度修正
```

IDF 记忆：

```text
一个词出现在越少的文档中，它的 IDF 越高；
一个词出现在越多的文档中，它的 IDF 越低。
```

使用流程：

```text
准备文档 → 文档分词 → 创建BM25 → 查询分词 → 计算得分 → 排序返回
```
